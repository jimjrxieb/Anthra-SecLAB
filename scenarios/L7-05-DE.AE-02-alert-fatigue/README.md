# L7-05 — DE.AE-02: Alert Fatigue (No Custom Falco Tuning)

## Scenario Summary

Falco is running. Alerts are firing. And that is exactly the problem.

The `anthra` cluster runs Falco with factory default rules — the same ruleset that
ships for every workload, every environment, every deployment pattern. The Portfolio
application runs health check probes every 10 seconds. Init containers install
dependencies with pip. Readiness probes curl internal endpoints. None of these are
threats. All of them trigger Falco rules.

The result: 400-600 Falco alerts per hour, of which more than 95% are false positives
generated by known-good processes. The L1 analyst opens the Falcosidekick dashboard
on Monday morning, sees the flood, and does what every analyst in this situation
eventually does — stops looking.

That is the real vulnerability. Not a misconfiguration of a single control. Not a
missing tool. The vulnerability is analyst behavior shaped by system noise. When
every alert is routine, no alert gets attention. The one real threat in the next 500
events is invisible — not because the detection engine missed it, but because the
analyst learned to ignore the output.

This scenario has no dramatic break step. The break already exists. Default Falco
rules in a production-like Kubernetes environment produce noise at a rate that
defeats the purpose of having a detection layer. The fix is SIEM tuning: creating
exception rules for known-good processes and writing custom rules that target
threats specific to the Portfolio application.

## What Is Already Broken

Falco runs with the default community ruleset. That ruleset was written for
general-purpose detection across all workloads. It did not account for:

- Portfolio health check probes calling `curl` inside the `api` container every 10s
- Readiness probes executing `ls` and reading `/proc` entries
- Init containers running `pip install` (spawning child processes, writing to tmp)
- ChromaDB startup scanning `/etc` configuration files
- Falco's own `package-audit-rule` firing on every apt/pip invocation

Each of these generates at least one alert. At 10-second probe intervals across
3 container types in 2+ replicas, the math produces:

- 360+ alerts/hour from health checks alone
- 50-100 alerts/hour from init container dependency installation
- 20-40 alerts/hour from legitimate file reads (config loading, lib scanning)

Total observed: 430-600 alerts/hour. True positive rate: under 5%.

What is missing on the other side: there are no custom rules for Portfolio-specific
threat patterns. A shell spawned inside the API container — the exact indicator of a
compromised application — fires the same generic `Terminal shell in container` rule
as every other exec, buried in the same flood.

## Why This Matters

NIST AU-6 requires that audit records be reviewed, analyzed, and reported in
accordance with defined frequencies and for specific purposes. A system generating
500 alerts per hour with a 95% false positive rate does not satisfy AU-6 — it
satisfies the letter while destroying the intent. Reviewers stop reviewing because
reviewing produces no signal.

CIS 8.11 requires that security event alert thresholds be tuned. This is not
optional. It is a named control requirement. Operating on default thresholds in a
production environment is a direct CIS 8.11 gap.

For DE.AE-02, the CSF requirement is that potentially adverse events be analyzed to
characterize them. You cannot characterize events you have trained yourself to ignore.
Alert fatigue is the mechanism by which high-volume noise converts to zero analysis.
The control fails in practice even when it functions technically.

In a FedRAMP Moderate environment, untuned SIEM alerting would appear in the AU
family gap analysis. The auditor asks not just "do you have alerting?" but "do you
review it?" Analyst behavior under noise conditions is evidence of whether AU-6 is
implemented or just installed.

## CySA+ OBJ 2.3 Teaching Point

Alert triage and SIEM tuning is a core CySA+ objective. This scenario demonstrates
the exact failure mode the exam tests: a detection control that is technically
operational but analytically useless due to noise volume.

The lesson is not "reduce alerts at all costs." The lesson is: every rule in your
ruleset should have a known true-positive rate, a known false-positive population,
and a documented justification for staying active. Rules that cannot meet that bar
are either suppressed with explicit exceptions or replaced with targeted alternatives.

Signal-to-noise ratio is an engineering problem. Treat it like one.
